# Counter Companion — desktop app for true realtime data

**Updated:** 2026-05-03
**Status:** Phase 0 (in progress)

The Counter webapp is a thin client. Counter Companion is the desktop process that lives on the player's machine, reads the only realtime data source Riot exposes (`127.0.0.1:2999/liveclientdata`), and pushes it to our cloud over WSS. The webapp/mobile view subscribes via SSE.

## Why a desktop companion at all

See `docs/research-realtime.md` for the full picture. Short version: live in-game state is only physically available on the player's own machine. Cloud-only realtime is impossible on Riot's API surface in 2026.

## Architecture

```
                   ┌──────────────────────────┐
                   │  Player's machine        │
                   │                          │
                   │  League of Legends       │
                   │   ↓ (local API)          │
                   │  https://127.0.0.1:2999  │
                   │   ↓ (poll @ 1-2 Hz)      │
                   │  Counter Companion       │
                   │   (Node, Electron later) │
                   └─────────────┬────────────┘
                                 │  WSS + companionToken
                                 ▼
                   ┌──────────────────────────┐
                   │  Vercel relay            │
                   │   /api/companion/ingest  │
                   │   in-memory keyed store  │
                   │   /api/companion/stream  │
                   └─────────────┬────────────┘
                                 │  SSE
                                 ▼
                   ┌──────────────────────────┐
                   │  Web/mobile client       │
                   │   /live/league/[id]      │
                   │   subscribes by token    │
                   └──────────────────────────┘
```

## Pairing flow

User-friendly, single device, no third-party OAuth:

1. Web view shows a 6-character pairing code (e.g. `K3F-9PD`) generated by `/api/companion/pair`. Code stored server-side with 5-min TTL.
2. User opens Companion → "Pair" → pastes the code.
3. Companion calls `POST /api/companion/claim` with the code, gets back a long-lived `companionToken`.
4. Companion stores the token in `~/.counter/companion.json`. Pairing complete.
5. From then on, every frame sent by Companion includes `Authorization: Bearer <companionToken>` header.

A given web client subscribes to `/api/companion/stream?token=<companionToken>`. We do NOT pin tokens to one match — the same token survives across all the user's games.

## Ingest format

Companion sends the raw `/allgamedata` JSON, plus a small envelope:

```ts
interface CompanionFrame {
  capturedAt: number;          // ms epoch on companion machine
  gameId: "league" | "dota";
  source: "live-client" | "gsi";
  payload: unknown;            // raw allgamedata or GSI body
}
```

Server normalises the payload into our existing `Match` shape via the adapter, runs the recommender, and stores the latest frame keyed by `companionToken`. SSE broadcasts the analyzed result, not the raw payload — keeps the wire format stable as we change companion internals.

## Phase 1 — distribution direction (recorded 2026-05-03)

The hard constraint stays: any customer-facing solution needs a process on
the player's machine that can reach `https://127.0.0.1:2999`. Options on the
table:

1. **Browser extension** (Chrome / Firefox / Edge). Lead candidate as of
   2026-05-03. Single-click install from the Web Store. No installer, no Apple
   Developer fee, no Windows EV cert, no notarization wait. Browser auto-updates.
   Chrome treats `127.0.0.1` as a secure context so the self-signed cert is
   not a blocker. Trade-off: only runs while the browser is open — but the
   user already has Counter open in the browser, so this is roughly free.
2. **Native desktop app** (Electron / Tauri / native). Higher fidelity (system
   tray, survives browser close, can auto-launch on login), but pays the full
   install + signing tax. Industry comp (Mobalytics, Blitz, LoLwiz) chose this
   path; they all predate the modern Web Store experience.
3. **Hybrid** — extension as the default zero-friction path, optional native
   app as a power-user upgrade.
4. **Stream-and-OCR** — user streams game to a private endpoint, cloud OCRs
   the HUD. Zero install. Massive engineering, lossy, fragile to UI patches,
   probably stretches Riot ToS. Not a serious candidate.

**No build plan committed yet.** Picking option 1, 2, or 3 happens before any
implementation work. The session that picks it is also the session that writes
the actual build plan.

## Compliance posture

**Counter is a side-window web/mobile app, not an in-game overlay.** That puts us in the same category as op.gg, mobalytics.gg (their websites), blitz.gg's web view — all of which show power spikes, lane matchups, build coaching, and live comp analysis without issue. Riot's strict feature blacklist (no enemy cooldowns, no ult timers, no action alerts) applies to *overlays rendered inside the game viewport*, not to a separate browser window the user voluntarily opens.

Two practical consequences:
- The full Match Intel suite (PowerSpikes, MacroCallBanner, ObjectiveTimers, LaneMatchups) belongs in the live view — that's the product. No demotion to post-game.
- The companion process surfaces nothing at all — it's a pure data pipe to our cloud.
- IF and ONLY IF we later wrap Counter in Overwolf for in-game overlay (Phase 3), THAT surface would have to obey Overwolf's compliance list. Not a Phase 0/1/2 concern.

The data we ever have via Live Client Data does not include enemy cooldowns, summoner-spell timers, or ult timers anyway — Riot doesn't expose those even to the local API for opponents. We're surfacing items + level + K/D/A/CS + position + scores, which is what every existing companion-website already shows.

## Security

- `companionToken` is a 256-bit random opaque string. No secrets baked into Companion binary.
- Cloud rate-limits ingest at 5 req/sec/token (well above 1-2 Hz polling).
- Companion only POSTs to our `/api/companion/ingest` over HTTPS. No other network access.
- We never receive Riot account credentials. Companion only sees what `/liveclientdata` already exposes to anything running on the player's machine.

## Files (Phase 0)

```
companion/
  package.json          # standalone npm package
  src/
    poll.ts             # polls 127.0.0.1:2999, pushes frames
    pair.ts             # one-time pairing flow
    config.ts           # token persistence (~/.counter/companion.json)
    log.ts              # tiny logger
  README.md             # how to run the dev script

src/app/api/companion/
  pair/route.ts         # POST → { code, expiresAt }
  claim/route.ts        # POST { code } → { token }
  ingest/route.ts       # POST { frame } with Bearer token
  stream/route.ts       # GET (SSE) ?token=...

src/lib/companion/
  store.ts              # in-memory frame store (Map<token, latest>)
  pair-codes.ts         # in-memory pairing-code store (5-min TTL)
  types.ts              # CompanionFrame, PairCode
```
